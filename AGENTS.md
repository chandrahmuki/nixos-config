# AGENTS.md — NixOS Config

> NixOS flake + Home Manager config for muggy-nixos. This file tells AI agents where things live, what conventions to follow, and which tools to use.

## What this project is

A declarative NixOS + Home Manager configuration for a desktop machine running niri (Wayland compositor), fish, neovim, and a curated set of apps. Built with flakes, SOPS-Nix for secrets, and Omnigraph for dependency tracking.

- **Host**: muggy-nixos | **User**: david | **Arch**: x86_64-linux
- **Deploy**: `nos` (alias for `nh switch`)
- **Entry point**: `flake.nix` → `nixosConfigurations.muggy-nixos`

## Architecture map

| Path | Responsibility |
|------|----------------|
| `flake.nix` | Flake entry point: inputs, outputs, host config |
| `flake.lock` | Pinned input revisions — commit after `nix flake update` |
| `home.nix` | Minimal HM user entry; modules handle everything else |
| `hosts/system/default.nix` | System config: bootloader, AMD GPU, greetd+niri, networking |
| `hosts/system/hardware-configuration.nix` | AUTO-GENERATED by nixos-generate, never edit |
| `modules/*.nix` | Auto-imported NixOS/HM modules via `scanModules` |
| `modules/default.nix` | Module scanner — reads all `.nix` files in `modules/` |
| `overlays.nix` | Package overrides (niri, opencode, parsec-bin, deno, openldap) |
| `lib/colors.nix` | Shared Tokyonight palette for theming |
| `memory/` | Session snapshots, lessons, skills, omnigraph graph DB |
| `.omnigraph/` | Dependency graph DB + HTML visualization |

### Module structure

Each `modules/<name>.nix` is a self-contained NixOS/HM module. New modules are auto-imported — just drop the file in `modules/`.

Common module pattern:
```nix
{ config, lib, pkgs, username, ... }:
{
  home-manager.users.${username} = { config, lib, ... }: {
    # HM config here
  };
}
```

### Runtime flow

1. `flake.nix` imports `modules/default.nix` which scans `modules/*.nix`
2. Each module contributes to `nixosConfigurations.muggy-nixos` or `home-manager.users.david`
3. `overlays.nix` patches specific packages via `nixpkgs.overlays`
4. `nos` (nh) builds and activates the new generation

## Conventions to follow

- **Module params**: system=`{config,lib,pkgs,...}` | HM=`{config,lib,pkgs,username,inputs,...}`
- **Never hardcode** `david` or `muggy-nixos` — use `username` / `hostname` args
- **Package refs**: `pkgs.jq` for derivations | `"${pkgs.pamixer}/bin/pamixer"` for binary paths | `inputs.<name>` for flake inputs
- **Colors**: `let colors = (import ../lib/colors.nix).tokyonight; in` then `colors.bg`, `colors.fg`, `colors.blue`
- **New modules**: `modules/<name>.nix` — auto-imported, no registration needed
- **Prefer** `settings` over `extraConfig` | use `lib.mkIf` / `lib.mkDefault` / `lib.mkForce`
- **xdg.configFile**: ne JAMAIS utiliser `force = true`. Utiliser `home-manager.backupFileExtension = "backup"` au niveau système pour gérer les conflits de fichiers.
- **2-space indent** | no comments unless asked | no emojis
- **Minimal diffs**: edit in place, no speculative abstractions

## OmniGraph

Omnigraph tracks module dependencies, sessions, lessons, and orphans. The graph DB lives at `.omnigraph/graph.db`.

### Commands
```
omnigraph check <file>       # Pre-edit: deps, sessions, lessons, risk
omnigraph query <term>       # Search nodes and edges
omnigraph orphans            # Detect unused inputs and dead refs
omnigraph session-resume     # Load last session context + summary
omnigraph lessons            # List all lessons
omnigraph impact <file>      # Blast radius of changing a file
omnigraph path <a> <b>       # Shortest path between two modules
omnigraph save               # Git commit + snapshot + rebuild graph (all-in-one)
```

### Workflow
- **Before editing** a `.nix` file: `omnigraph check <file>` to understand dependencies
- **Before editing** any file: `omnigraph query error` to review past mistakes on that file/module
- **Before editing** any file: `omnigraph impact <file>` to understand blast radius
- **After important changes**: `omnigraph save` to commit, snapshot, and rebuild graph
- **Periodically**: `omnigraph orphans` to clean up dead refs
- **Lessons**: stored as nodes with `lesson_applies_to` edges pointing to affected modules

### Error prevention workflow (mandatory)
1. `omnigraph check <file>` — check deps, sessions, lessons, risk
2. `omnigraph query error` — review past errors on this file
3. `omnigraph impact <file>` — understand what could break
4. Implement the change
5. `nix eval` to verify

### Graph schema
- **Nodes**: `file`, `input`, `option`, `session`, `lesson`, `skill`, `error`, `concept`, `function`
- **Edges**: `session_modified`, `session_produced_lesson`, `lesson_applies_to`, `uses_input`, `uses_module`, `imports`, `indexes`

## Skills to use

- **`/omnigraph`** — Query the dependency graph before editing modules. Always run `check` before changes, `session-resume` to restore context.
- **`/project-map`** — Regenerate the compact NixOS config project map. Use after adding new modules.

Golden path for a non-trivial change:
1. `omnigraph check <file>` to understand dependencies
2. `omnigraph query error` to review past mistakes
3. `omnigraph impact <file>` to understand blast radius
4. Plan the change (ask if unsure)
5. Implement the narrowest change that works
6. `nix eval` to verify
7. `omnigraph save` to commit, snapshot, and rebuild graph

## Rules

- Always ask before delete/modify/commit
- `git add` new files immediately (flake needs it)
- After `.nix` changes: run `nix eval` to verify
- Secrets: `modules/secrets.nix` uses SOPS-Nix — never commit plaintext secrets
- Every 15 messages: remind user to run `/compact` now to save tokens
- **Never touch omnigraph source code** — it's maintained by another agent. Only use the CLI commands.
- Never run `nos` yourself; always ask the user to run it instead.
- Toujours tester et valider le fonctionnement réel des configurations en arrière-plan (statut de service, logs, existence des fichiers, processus) avant de confirmer à l'utilisateur que tout est fonctionnel.

## Things to know about the maintainer's preferences

- Keep responses and commits terse; no trailing summaries unless asked
- French communication preferred
- No emojis in code or messages
- Prefer HM module patterns over standalone configs
- Overlay packages rather than waiting for nixpkgs updates when needed
- **Mesa Pinning**: Mesa system packages must remain pinned to version 26.1.2 (via `nixpkgs-mesa` input) to prevent multi-monitor Parsec XWayland decoding crashes. Check newer versions (e.g. 26.2+) during future upgrades.
